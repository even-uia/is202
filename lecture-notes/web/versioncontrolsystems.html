Version Control Systems

Version Control Systems (VCS), or Source Code Control Systems, are essential tools in all real-life system development projects. They provide a robust and secure way of sharing the source code between the developers. They fill the role of project archive, and in the archive (or repositories they are commonly called) there will be backups that a developer can fall back on after various disasters. It will also be possible to revert to a snapshot of the code made at important moments in time, e.g. a delivery to the customer.

What do vc systems do?

An important part of any vc systems is the repository. The repository is where the source code is stored. Then there is the client, which each developer will use to maintain his working copy. The working copy is the developers own copy the files from the repository. The client compares the working copy with the contents of the repository, and can update the repository with the changes made by the developer, or it can update the working copy with changes made by other members of the team.

When updating to or from the working copy, a conflict can occur. A conflict happens when two developers have made changes to the same file, that are so intertwined that the merging algorithm cannot cope.

When there are no conflicts, or all have been resolved, the update is accepted into the repository. The remembers all updates it receives, when they happened and what the changes were. So you can get back to any old version of the system, provided you remember when that version was merged into the repository. However this is a very useful operation, because it allows the team to recreate, for example, the latest published version of the system, in a couple of minutes, so it can be tested or debugged.

Choosing a version control system

There are two major types of version control systems: Centralized systems and distributed version control systems. Subversion and Git are the most widespread open-source version control systems of each type

Git is a distributed version control system. Distributed means that there is not just one repository. In fact every project member will have their own copy of the "master" repository. (Master in quotes, because having a master repository isn't necessary. It's just an agreement within the project that the contributions from all members shall be merged into a particular repository

GIT

The text below describes how to set up and use a git repository on GitHub. All group members must register as users of github. To register:

  • Go to https://github.com
  • Enter a username, you email, and password
  • Click on the green button at the bottom of the form

Creating a repository

The actions described under this heading should only be carried out once, by the repository owner (the group member who created it). You will not need more than one master repository.

  • Log in on github
  • Click the little '+' in the upper right corner of the page
  • Select new repository
  • In the new repository form enter a name for the repository. In the description field you may enter something like: my group's project repository"
  • Leave the rest as it is, and click on Create repository

Before you start using the repository it is a good idea to create new branch that will be your working branch. (Tip: Read the " GitHub Guide) You can create the branch in GitHubs web interface:

  • Go to the repository page on github.com
  • click on the Branch: master button, and fill in the name of the working branch
  • enter the name of the working branch in the dialog, and click on the Create [Branch name] button.

  • Enter something that will identify the new collaborator in the search box. Select the person you want to add in the search hitlist.
